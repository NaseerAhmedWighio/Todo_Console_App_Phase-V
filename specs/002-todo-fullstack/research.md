# Research: Todo Full-Stack Web Application - Phase II

## High-Level System Architecture

### Frontend: Next.js App Router
- **Technology**: Next.js 14+ with App Router
- **Features**: Server-side rendering, client-side rendering, static generation
- **Routing**: File-based routing system using the app directory
- **Styling**: CSS Modules, Tailwind CSS, or styled-components
- **State Management**: React Context API or Zustand for client-side state
- **API Communication**: fetch API or axios for backend communication

### Backend: FastAPI REST API
- **Technology**: FastAPI 0.104+ with Python 3.11+
- **Features**: Async/await support, automatic API documentation (Swagger/OpenAPI)
- **Database ORM**: SQLModel for database operations
- **Authentication**: Integration with Better Auth JWT tokens
- **Validation**: Pydantic for request/response validation
- **Error Handling**: Custom exception handlers

### Database: Neon PostgreSQL
- **Type**: Serverless PostgreSQL database
- **Connection**: Connection pooling and async drivers (asyncpg)
- **Migration Tool**: Alembic for schema migrations
- **ORM**: SQLModel for object-relational mapping

## Authentication Flow using Better Auth and JWT

### Frontend Authentication Flow
1. User visits the application
2. Better Auth handles registration/login on frontend
3. JWT token is stored securely (httpOnly cookie or secure localStorage)
4. JWT token is sent with each API request in Authorization header
5. Frontend handles token expiration and refresh

### Backend Authentication Flow
1. API request received with Authorization header
2. JWT token extracted and verified using Better Auth public key
3. User identity extracted from token payload
4. Request processed with user context
5. Unauthorized requests return 401 status code

### Token Structure
- **Access Token**: Short-lived (15-30 minutes)
- **Refresh Token**: Long-lived (7-30 days) - stored as httpOnly cookie
- **Claims**: user ID, email, roles, expiration time

## API Request Lifecycle from Frontend to Backend

### 1. Client Request Initiation
- Frontend component triggers API call
- Authorization header with Bearer token added automatically
- Request body validated before sending

### 2. Network Transport
- HTTPS encryption for all API requests
- Standard HTTP methods (GET, POST, PUT, DELETE)
- Proper content-type headers (application/json)

### 3. Backend Processing
- Request received by FastAPI
- Authentication middleware verifies JWT token
- Request validation using Pydantic models
- Business logic executed in service layer
- Database operations using SQLModel

### 4. Response Handling
- Response formatted as JSON
- Proper HTTP status codes returned
- Error responses follow consistent format
- Response sent back to frontend

## Database Schema Overview

### Users Table
- `id`: UUID primary key
- `email`: String (unique, not null)
- `name`: String (optional)
- `created_at`: DateTime (default: now)
- `updated_at`: DateTime (default: now, auto-update)

### Tasks Table
- `id`: UUID primary key
- `title`: String (not null)
- `description`: Text (optional)
- `completed`: Boolean (default: false)
- `user_id`: UUID foreign key (references users.id)
- `created_at`: DateTime (default: now)
- `updated_at`: DateTime (default: now, auto-update)

### Indexes
- Users: email (unique index)
- Tasks: user_id (index for efficient user filtering)
- Tasks: (user_id, completed) (composite index for common queries)

## Environment Variable Strategy

### Backend Environment Variables
- `DATABASE_URL`: Connection string for Neon PostgreSQL
- `BETTER_AUTH_SECRET`: Secret key for JWT signing verification
- `BETTER_AUTH_URL`: Base URL for Better Auth service
- `ACCESS_TOKEN_EXPIRE_MINUTES`: Token expiration time (default: 30)
- `ALGORITHM`: JWT algorithm (default: HS256)

### Frontend Environment Variables
- `NEXT_PUBLIC_API_BASE_URL`: Base URL for backend API
- `NEXT_PUBLIC_BETTER_AUTH_URL`: Base URL for Better Auth

### Security Considerations
- Never expose database credentials to frontend
- Use httpOnly cookies for sensitive tokens when possible
- Environment files should be git-ignored
- Use .env.example for documentation of required variables

## Deployment Separation between Frontend and Backend Services

### Backend Deployment
- Deployed as independent service on platforms like:
  - Railway
  - Render
  - Vercel (for serverless functions)
  - AWS/Azure/GCP
- Environment variables configured per deployment environment
- Database migrations run during deployment
- Health check endpoints for monitoring

### Frontend Deployment
- Deployed as separate application on platforms like:
  - Vercel
  - Netlify
  - Cloudflare Pages
- Environment variables for API base URL configured per environment
- Static asset optimization and CDN distribution
- Client-side routing handled by Next.js

### Integration Points
- API base URL configured as environment variable
- CORS settings configured to allow frontend domain
- Authentication flow works across separate deployments
- Separate monitoring and logging for each service
