# Implementation Plan: Advanced Todo Features - Phase III

**Branch**: `003-advanced-todo-features` | **Date**: 2026-02-19 | **Spec**: [spec.md](spec.md)

## Summary

Implementation of advanced task management features including recurring tasks, due dates with reminders, priorities, tags, search, filtering, and sorting. The system architecture is enhanced with event-driven capabilities using Apache Kafka for event streaming and Dapr for distributed application runtime. The implementation extends the existing Phase II full-stack application (Next.js + FastAPI + SQLModel + Neon PostgreSQL) while maintaining backward compatibility and user isolation.

## Technical Context

**Language/Version**: Python 3.11 (Backend), TypeScript/JavaScript (Frontend)
**Primary Dependencies**: FastAPI, Next.js App Router, SQLModel, Apache Kafka, Dapr, Neon PostgreSQL
**Storage**: Neon Serverless PostgreSQL with SQLModel ORM, Kafka for event streaming
**Testing**: pytest (Backend), Jest/React Testing Library (Frontend)
**Target Platform**: Web application (cross-platform)
**Project Type**: Web (separate frontend and backend applications)
**Performance Goals**: 
- Search results in under 3 seconds for 10,000 tasks per user
- Filter/sort operations complete in under 1 second for 1,000 tasks
- Event processing latency under 100ms at p95
- Reminder delivery within 1 minute of scheduled time with 99% reliability
**Constraints**: 
- Maintain backward compatibility with Phase II features
- All task operations must remain user-scoped (no cross-user access)
- JWT authentication required for all API requests
- Stateless backend architecture (no in-memory state)
**Scale/Scope**: 
- Support 10,000 tasks per user
- Horizontal scaling with Dapr for 10x current load
- 99.9% uptime for core features during event processing

## Complexity Tracking

> **GATE: All violations must be justified before proceeding**

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Apache Kafka | Event-driven architecture required for scalability, real-time updates, and future integrations; supports event retention for unavailable consumers | Direct database polling rejected: doesn't scale, violates stateless backend principle, introduces latency |
| Dapr runtime | Distributed application runtime provides production-ready state management, pub/sub, service discovery; enables horizontal scaling | Custom microservices communication rejected: reinvents wheel, error-prone, lacks production testing |
| Background workers | Recurring task generation and reminder delivery require scheduled background processing independent of user requests | Cron jobs rejected: not scalable, single point of failure, hard to distribute across instances |
| Search service | Full-text search across 10,000+ tasks requires indexed search for performance | Simple SQL LIKE queries rejected: too slow for large datasets, no relevance ranking |

## Phase 0: Research & Discovery

### Research Tasks

1. **Kafka Integration Pattern**: Research best practices for integrating Apache Kafka with FastAPI applications
   - Kafka producer/consumer setup in Python
   - Event schema design and versioning
   - Error handling and retry strategies
   - Kafka connection pooling and performance

2. **Dapr State Management**: Research Dapr state management patterns for Python applications
   - State store configuration for PostgreSQL
   - Transactional state operations
   - State consistency models
   - Dapr sidecar communication patterns

3. **Recurring Task Algorithms**: Research algorithms for recurring date calculation
   - Handling edge cases (month-end dates, leap years)
   - RFC 5545 iCalendar recurrence rules
   - Timezone-aware recurrence
   - Performance optimization for bulk generation

4. **Reminder Delivery Systems**: Research reminder notification delivery patterns
   - Email service integration (SendGrid, AWS SES, etc.)
   - Web push notification standards
   - In-app notification patterns
   - Retry strategies for failed deliveries

5. **Full-Text Search in PostgreSQL**: Research PostgreSQL full-text search capabilities
   - tsvector/tsquery optimization
   - Search ranking and relevance
   - Index strategies for performance
   - Multi-column search patterns

6. **Tag Management Systems**: Research tag/folksonomy system design patterns
   - Many-to-many tag relationships
   - Tag autocomplete and suggestions
   - Tag merging and cleanup
   - Performance optimization for tag queries

### Research Findings Summary

**Decision**: Use kafka-python library for Kafka integration
- **Rationale**: Mature, well-documented, async support, integrates well with FastAPI
- **Alternatives**: aiokafka (async-first, but less mature), confluent-kafka (faster but requires C library)

**Decision**: Use Dapr Python SDK for state management and pub/sub
- **Rationale**: Official SDK, simple API, handles sidecar communication automatically
- **Alternatives**: Direct HTTP to Dapr API (more verbose, error-prone)

**Decision**: Implement custom recurrence engine based on RFC 5545 patterns
- **Rationale**: Full iCalendar library overkill; custom implementation handles our specific needs
- **Alternatives**: python-dateutil.rrule (good but complex), recurrence (lightweight, less features)

**Decision**: Use PostgreSQL full-text search with tsvector indexes
- **Rationale**: No additional infrastructure, good performance, integrates with existing DB
- **Alternatives**: Elasticsearch (overkill for our scale), Algolia (external dependency, cost)

**Decision**: Implement tag system with many-to-many relationship tables
- **Rationale**: Standard pattern, efficient queries, supports multiple tags per task
- **Alternatives**: JSON array (harder to query), separate tag tables per user (complex)

**Decision**: Background workers using Celery with Redis broker
- **Rationale**: Mature, scalable, integrates with FastAPI, supports scheduled tasks
- **Alternatives**: APScheduler (simpler but less scalable), RQ (Redis-only, less features)

## Phase 1: Design & Contracts

### Data Model Extensions

See: `data-model.md` (generated in same phase)

**New Entities**:
- RecurringTask: Extends Task with recurrence_pattern, frequency, end_condition, series_id
- Reminder: id, task_id, timing_minutes, timing_days, delivery_channel, sent_at
- Tag: id, user_id, name, color, created_at
- TaskTag: Many-to-many join table (task_id, tag_id)
- DomainEvent: id, event_type, payload, timestamp, user_id, processed

**Extended Entities**:
- Task: Add priority (enum), due_date (datetime), timezone (string)
- User: Add timezone (string), notification_preferences (json)

### API Contracts

See: `contracts/` directory (generated in same phase)

**New Endpoints**:
- `POST /api/v1/todos/{id}/recurring` - Configure recurring pattern
- `GET /api/v1/tags` - List user's tags
- `POST /api/v1/tags` - Create tag
- `DELETE /api/v1/tags/{id}` - Delete tag
- `GET /api/v1/search` - Search tasks
- `GET /api/v1/reminders` - List reminders
- `POST /api/v1/reminders` - Create reminder
- `DELETE /api/v1/reminders/{id}` - Delete reminder

**Extended Endpoints**:
- `POST /api/v1/todos` - Add priority, due_date, tags fields
- `PUT /api/v1/todos/{id}` - Support updating advanced fields
- `GET /api/v1/todos` - Add query params: filter, sort, search

### Quick Start Guide

See: `quickstart.md` (generated in same phase)

**Prerequisites**:
- Docker & Docker Compose
- Existing Phase II setup complete
- OpenRouter API key configured

**Setup Steps**:
1. Start Kafka & Dapr services: `docker-compose up -d kafka dapr`
2. Run database migrations: `cd backend && python -m sqlmodel_migrate upgrade`
3. Install new dependencies: `pip install -r requirements.txt`
4. Configure Dapr components: `dapr init`
5. Start backend with Dapr: `dapr run --app-id todo-backend -- python main.py`
6. Start frontend: `cd frontend && npm run dev`

**Verification**:
- Create a recurring task → verify new instance auto-generated
- Set a reminder → verify notification delivered at scheduled time
- Search for tasks → verify results appear instantly
- Apply filters/sorts → verify UI updates correctly
